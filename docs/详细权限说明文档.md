# 详细权限说明文档

## 1. 目标与原则

本平台权限设计目标：

- 支持公司内部多角色协同，保证信息透明与可追踪。
- 区分“身份标签”和“系统权限角色”，一个人可多身份、多角色。
- 同时支持板块级权限与项目级权限。
- 对保密项目提供可控访问范围（不是全员可见）。

设计原则：

- 功能权限：采用 RBAC（用户-角色-权限）。
- 数据可见范围：采用项目级授权与保密级别控制。
- 可审计：关键权限变更、项目成员变更留痕。

---

## 2. 用户信息模型（建议）

用户基础字段应包含：

- 用户名（唯一，用于登录）
- 姓名（展示名）
- 手机号（可选登录账号）
- 状态（active/disabled）
- 身份标签（可多选）
- 系统角色（可多选）

### 2.1 身份标签（Identity Tags）

作用：用于业务身份标识、筛选和协作分工，不直接决定系统权限。

建议标签：

- 公司领导
- 项目经理
- 前端开发
- 后端开发
- 算法工程师
- 测试工程师
- UI设计师
- 运维工程师
- 商务经理

说明：

- 一个用户可拥有多个标签（例如：项目经理 + 后端开发）。
- 标签用于任务分配、人员筛选、统计分析。

### 2.2 系统角色（Roles）

作用：用于决定可访问哪些板块、可执行哪些系统操作（RBAC）。

建议角色：

- `admin`：系统管理员，拥有全局管理能力。
- `project-manager`：项目管理角色，可创建/管理项目内业务对象（按范围）。
- `developer`：研发协作角色，可参与项目任务流转。
- 后续可扩展：`leader`、`qa`、`ops`、`business` 等。

说明：

- 一个用户可拥有多个角色。
- 最终权限为所有角色权限并集。

---

## 3. 权限分层

## 3.1 板块权限（Module Permission）

定义：用户可访问系统哪些一级/二级功能板块。

示例板块：

- dashboard
- project
- requirement
- task
- test
- collaboration
- okr-report
- system

控制方式：

- 角色绑定板块权限。
- 登录后后端根据用户角色计算可见菜单并返回前端。

## 3.2 项目权限（Project Permission）

定义：用户在具体项目中的数据可见性与操作能力。

关键维度：

- 是否可见该项目
- 项目内角色（owner/manager/member/observer 等）
- 在项目中的可操作范围（创建任务、评论、提醒、上传附件等）

建议规则：

- 非项目成员默认不可见项目详情。
- 项目成员可见项目数据并参与协作。
- 项目负责人/项目管理员可调整项目成员与项目权限。

---

## 4. 保密项目设计

## 4.1 项目保密级别（建议）

为项目增加 `confidential_level` 字段，建议枚举：

- `L0_PUBLIC`：公开（组织内默认可见）
- `L1_INTERNAL`：内部（需登录，默认按部门/组织可见）
- `L2_RESTRICTED`：受限（仅项目成员可见）
- `L3_CONFIDENTIAL`：保密（仅明确授权用户可见）

## 4.2 访问控制规则

建议判定顺序：

1. `admin` 可见所有项目。
2. `L0/L1` 按组织规则可见（可结合部门策略）。
3. `L2/L3` 必须是项目成员或被授权用户才可见。
4. 被移出成员后立即失去项目访问。

## 4.3 项目安排与人员选择

在创建/编辑项目时增加：

- 保密级别选择
- 可见成员名单（白名单）
- 项目角色分配（负责人、经理、成员、观察者）

---

## 5. 用户管理页应支持的权限相关字段

用户管理页建议展示与编辑：

- 用户名
- 姓名
- 手机号
- 身份标签（多选）
- 系统角色（多选）
- 板块权限预览（由角色计算）
- 项目权限摘要（参与项目数、保密项目数、项目角色分布）
- 状态（启用/禁用）

---

## 6. 权限校验实现建议（与当前架构兼容）

后端（FastAPI）：

- 登录态：JWT
- 角色权限：RBAC（User-Role-Permission）
- 板块权限：`permission.code`
- 项目权限：在项目查询与详情接口统一做成员/保密校验

接口层建议：

- 路由级校验：是否登录、是否 superuser、是否拥有某板块权限
- 服务层校验：是否有该项目访问权、是否有该项目内操作权限

---

## 7. 审计与追踪要求

需要记录日志的权限相关操作：

- 用户角色变更
- 用户身份标签变更
- 用户状态变更（禁用/启用）
- 项目保密级别变更
- 项目成员增删与项目角色调整
- 特殊授权（加入/移出保密项目）

---

## 8. 迭代建议

第一阶段（当前）：

- 完成身份标签字段与管理页面编辑
- 完成项目保密级别字段与可见性控制
- 完成项目成员授权可见

第二阶段：

- 细化项目内权限（仅查看、可编辑任务、可管理成员等）
- 增加部门级数据范围策略

第三阶段：

- 增加策略引擎（可配置化权限规则）
- 完善审计报表与异常访问告警

## 9. 最新补充（2026-03-21 00:34:56 +08:00）

1. 项目创建权限调整
- 平台支持“公司项目管理 + 员工个人项目管理”双场景。
- 所有登录用户都可以创建项目，不再限制为管理员专属。

2. 项目归属字段
- 项目新增 `project_type` 字段，用于区分：
- `work`：公司项目
- `personal`：个人项目

3. 默认展示与筛选规则
- “项目清单 / 任务清单”默认展示当前用户参与项目，且默认优先展示进行中项目。
- 新增“公司项目”板块，用于查看当前用户权限范围内的公司项目清单。
- 新增“排除个人项目（exclude_personal）”筛选项，便于会议、周报等场景一键过滤个人项目。

4. 权限与可见性边界
- 个人项目默认由创建人主导管理，可按成员机制共享给协作者。
- 公司项目按项目成员与保密级别规则控制可见范围。
- 管理员仍保留全局可见与治理能力，用于审计与应急处理。
