# 性能优化与容错说明（2026-03-20）

## 1. 背景问题

当项目、任务、评论数据量增长后，系统出现两类问题：

1. 查询慢：列表页响应时间显著增加。
2. 体验差：某个接口卡顿会导致页面整体“无响应”。

## 2. 根因分析

### 2.1 后端查询层

1. 列表接口曾采用“先全量查询，再 Python 切片分页”，导致数据库与应用层都承担了不必要开销。
2. 列表查询加载了大量仅详情页才需要的关联数据（成员、节点、评论、附件、状态历史等）。
3. 标签接口复用了重型项目查询链路，获取标签成本过高。
4. 高频筛选字段和关联表缺少足够索引。

### 2.2 前端渲染层

1. 页面根节点存在全局 `v-loading`，当任一请求慢或失败时会遮罩整个页面。
2. 初始化采用 `Promise.all`，任一接口失败会中断整体初始化流程。

## 3. 已落地优化

## 3.1 后端优化

1. 项目清单改为数据库分页（`LIMIT/OFFSET + COUNT`）。
2. 任务清单改为数据库分页，并返回 `total/page/page_size`。
3. 项目、任务查询拆分为：
   - 列表轻查询（仅加载列表所需字段与轻量关联）
   - 详情重查询（加载评论、附件、状态历史等）
4. 标签接口改为仅查询 `Project.tags` 字段。
5. 初始化阶段补充索引：
   - `projects(status, priority, owner_id, created_at)`
   - `tasks(project_id, parent_task_id, status, priority, assignee_id)`
   - `project_members(project_id, user_id)`
   - `project_watchers(project_id, user_id)`
   - `task_watchers(task_id, user_id)`

## 3.2 前端容错优化

1. 项目详情页取消根节点全局 `v-loading`，改为板块级 loading：
   - 项目信息
   - 任务列表
   - 操作日志
   - 任务详情抽屉
2. 初始化请求改为 `Promise.allSettled`，单模块失败不影响其他模块可用。
3. 关键加载函数增加独立错误处理，失败只提示当前模块，不阻塞整体交互。

## 4. 预期效果

1. 大数据场景下，清单页首屏与翻页性能更稳定。
2. 某个接口异常时，页面其余区域仍可继续使用，避免“整页卡死”。

## 5. 后续建议

1. 增加慢查询日志（SQL 耗时阈值告警）。
2. 增加基准脚本，对比优化前后耗时与吞吐。
3. 对热点列表接口增加请求取消与防抖，避免频繁筛选导致请求堆积。
