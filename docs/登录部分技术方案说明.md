# 登录部分技术方案说明

## 1. 文档目的

本文档用于说明当前项目中“登录、登录态校验、菜单权限管理、用户权限配置”这一部分所采用的技术选型与实现方案。

重点回答以下问题：

- 登录功能使用了 FastAPI 的哪些组件
- 权限管理当前是如何实现的
- 为什么当前阶段采用这种实现方式
- 后续正式开发阶段如何演进

## 2. 当前技术选型结论

当前登录与权限管理部分，没有引入额外的完整认证框架，例如：

- `fastapi-users`
- `fastapi-login`
- `Authlib`
- `Casbin`

而是采用了 **FastAPI 原生组件 + JWT + SQLite + 基础 RBAC 模型** 的方式。

当前核心技术选型如下：

1. Web 框架：`FastAPI`
2. 路由组织：`APIRouter`
3. 请求/响应校验：`Pydantic`
4. 登录态校验：`Depends + Header`
5. 登录令牌：`JWT Bearer Token`
6. 密码加密：`passlib` 的 `pbkdf2_sha256`
7. 用户与权限逻辑：自定义 `AuthService`
8. 权限模型：`User + Role + Permission + 关系表`
9. 菜单权限模型：自定义 `MENU_TREE + RBAC 权限映射`

这意味着当前方案是：

- 不依赖第三方完整权限框架
- 使用 JWT
- 不依赖 OAuth2
- 使用 SQLite 持久化用户、角色、权限
- 以本地开发和功能快速打通为目标

## 3. 当前代码中使用到的 FastAPI 组件

## 3.1 `APIRouter`

用于组织登录和用户管理相关接口。

当前相关文件：

- [auth.py](/d:/py6/ProjectTrace/backend/app/api/v1/endpoints/auth.py)
- [users.py](/d:/py6/ProjectTrace/backend/app/api/v1/endpoints/users.py)
- [router.py](/d:/py6/ProjectTrace/backend/app/api/v1/router.py)

用途：

- 将登录接口单独放在 `/auth`
- 将用户管理接口单独放在 `/users`
- 通过统一版本路由 `/api/v1` 管理接口

## 3.2 `Depends`

用于实现登录态依赖注入。

当前相关文件：

- [security.py](/d:/py6/ProjectTrace/backend/app/core/security.py)
- [auth.py](/d:/py6/ProjectTrace/backend/app/api/v1/endpoints/auth.py)
- [users.py](/d:/py6/ProjectTrace/backend/app/api/v1/endpoints/users.py)

用途：

- 在访问受保护接口时自动校验请求头中的登录信息
- 如果未登录或 token 无效，则直接返回 401

这是当前权限校验的核心入口。

## 3.3 `Header`

在 `get_current_user` 中通过 `Header` 读取 `Authorization` 请求头。

当前实现方式：

- 前端请求时携带 `Authorization: Bearer <token>`
- 后端从 Header 中提取 token
- 调用 `auth_service.get_user_by_token(db, token)` 校验登录态

这是一种非常典型的 Bearer Token 认证方式。

## 3.4 `HTTPException`

用于返回认证失败、参数错误、资源不存在等异常。

当前使用场景：

- 用户名/手机号或密码错误时返回 `401`
- 登录态无效时返回 `401`
- 用户不存在时返回 `404`
- 用户名或手机号重复时返回 `400`

## 3.5 `Pydantic Schema`

用于定义登录请求、登录响应、用户新增、用户修改、用户返回结构。

当前相关文件：

- [auth.py](/d:/py6/ProjectTrace/backend/app/schemas/auth.py)
- [user.py](/d:/py6/ProjectTrace/backend/app/schemas/user.py)

用途：

- 校验登录输入参数
- 约束用户字段结构
- 保证接口返回结构统一

## 4. 当前登录方案实现方式

## 4.1 登录接口

当前登录接口：

- `POST /api/v1/auth/login`

支持两种登录方式：

- 用户名 + 密码
- 手机号 + 密码

实现逻辑：

1. 前端提交 `account` 和 `password`
2. 后端在 `AuthService` 中查找用户
3. 若 `account` 匹配 `username` 或 `mobile`，且密码正确，则登录成功
4. 后端校验数据库中的密码哈希
5. 后端生成 JWT
6. 返回：
   - `access_token`
   - `token_type`
   - `user`
   - `menus`

对应实现文件：

- [auth.py](/d:/py6/ProjectTrace/backend/app/api/v1/endpoints/auth.py)
- [auth_service.py](/d:/py6/ProjectTrace/backend/app/services/auth_service.py)

## 4.2 Token 生成机制

当前 token 生成方式：

- 使用 `python-jose` 生成 JWT
- 当前签名算法：`HS256`
- 当前 token 中包含：
  - `sub`：用户 ID
  - `exp`：过期时间

特点：

- 支持签名
- 支持过期时间
- 适合前后端分离系统
- 能够平滑升级到正式方案

缺点：

- 当前还没有 Refresh Token
- 当前还没有 Redis 黑名单 / 会话撤销机制

## 4.3 登录态保存方式

当前登录态采用：

- 前端持有 JWT
- 后端在每次请求时解析 JWT
- 根据 JWT 中的 `sub` 字段查找 SQLite 中的用户
- 不再使用内存 token 映射

优点：

- 不依赖内存会话
- 服务重启后 JWT 仍可验证
- 更接近正式系统方案

缺点：

- 暂未实现 Refresh Token
- 暂未实现 Redis 会话统一失效控制

## 5. 当前权限管理方案

## 5.1 权限模型

当前权限模型采用基础 RBAC：

- 用户 `User`
- 角色 `Role`
- 权限 `Permission`
- 用户角色关系 `user_roles`
- 角色权限关系 `role_permissions`

但权限粒度目前仍以“板块可使用范围”为主。

也就是：

- 用户拥有某个角色
- 角色拥有一组板块权限
- 就可以访问该板块下的一、二级菜单

这与当前项目阶段是匹配的，因为你现在要求的是：

- 先完成登录
- 先完成系统目录骨架
- 先完成用户权限配置
- 权限重点是“哪个板块可以使用”

## 5.2 菜单定义方式

当前菜单结构由代码中的 `MENU_TREE` 统一定义。

对应文件：

- [menu_service.py](/d:/py6/ProjectTrace/backend/app/services/menu_service.py)

一级菜单示例：

- 工作台
- 项目管理
- 需求管理
- 任务管理
- 测试缺陷
- 工时协同
- OKR与报表
- 系统管理

二级菜单示例：

- 项目中心
- 项目总览
- 需求中心
- 需求池
- 任务看板
- 任务日历
- 用户管理
- 权限说明

## 5.3 权限字段设计

当前数据库设计中核心关系为：

- `users`
- `roles`
- `permissions`
- `user_roles`
- `role_permissions`

用户最终可访问的板块权限，是由其所属角色汇总计算出来的。

当前用户核心字段包括：

- `username`
- `real_name`
- `mobile`
- `password_hash`
- `status`
- `is_superuser`

## 5.4 菜单权限生成方式

当前通过：

- `Role -> Permission`
- `Permission code -> MENU_TREE`
- `build_menu_tree(module_permissions)`

根据用户拥有的板块权限过滤菜单树，返回前端真正可见的菜单。

这使得：

- 前端不需要自己硬编码判断权限
- 后端统一返回当前用户可用菜单
- 菜单与权限来源一致

## 5.5 用户权限管理接口

当前用户管理接口：

- `GET /api/v1/users`
- `POST /api/v1/users`
- `PUT /api/v1/users/{user_id}`
- `DELETE /api/v1/users/{user_id}`
- `GET /api/v1/users/meta/modules`
- `GET /api/v1/users/meta/roles`

其中：

- `meta/modules` 用于返回所有可配置的板块权限列表
- `meta/roles` 用于返回所有可分配角色
- 新增、修改用户时通过 `role_ids` 分配角色

## 6. 当前方案为什么这样选

当前采用 FastAPI 原生能力 + JWT + SQLite + 基础 RBAC，而不是直接引入更重的认证/权限框架，主要原因如下：

### 6.1 更适合当前开发阶段

当前项目仍在：

- 产品设计阶段向开发阶段过渡
- 功能在快速落地
- 数据库模型还未完全定稿

这时引入复杂认证框架反而会增加理解和接入成本。

### 6.2 方案 B 比方案 A 更规范，但仍保持可控复杂度

相比之前的方案 A：

- 现在已经引入 JWT
- 已经引入 SQLite 持久化
- 已经引入基础 RBAC 关系模型

但仍然没有直接引入：

- 完整 OAuth2 平台
- Casbin 复杂授权规则
- 外部身份中心

这样可以在规范性和开发速度之间取得平衡。

### 6.3 当前权限需求相对简单

现在的权限需求主要是：

- 谁能登录
- 谁能访问哪个业务板块

并不需要：

- OAuth 第三方登录
- 社交登录
- SSO
- 复杂 ABAC
- 多租户隔离
- 复杂审计审批权限

### 6.4 便于后续平滑演进

当前方案虽然仍然偏轻量，但结构上已经做了职责分层：

- 接口层：`endpoints`
- 校验层：`schemas`
- 安全入口：`security.py`
- 业务逻辑层：`auth_service.py`
- 用户查询与序列化层：`user_service.py`
- 菜单权限层：`menu_service.py`
- RBAC 组合逻辑层：`rbac_service.py`
- RBAC 数据层：`User / Role / Permission`
- 初始化种子数据：`init_db.py`

后面切换到 PostgreSQL、Refresh Token、Redis Session、细粒度 RBAC，都可以在这个基础上渐进替换。

## 7. 当前方案的优点

- 比纯内存方案更规范
- JWT 更适合前后端分离
- SQLite 足够支撑当前开发阶段
- RBAC 基础模型已经建立
- 后续切换 PostgreSQL 成本更低
- 和当前“板块级权限”需求仍然保持匹配

## 8. 当前方案的不足

当前方案只适合开发阶段和早期原型阶段，不适合直接作为正式生产方案。

主要不足：

1. 当前数据库仍是 SQLite，适合开发，不适合高并发生产
2. 权限粒度还只是板块级
3. 没有刷新 token 机制
4. 没有 Redis 会话统一失效能力
5. 没有登录失败限制
6. 没有细粒度接口权限和数据范围权限

## 9. 后续推荐演进方案

## 9.1 数据持久化

当前已经具备：

- 用户表
- 角色表
- 权限表
- 用户角色关系表
- 角色权限关系表

下一步建议继续完善：

- 用户状态审计字段
- 登录日志
- 刷新 token 表
- 密码修改记录

开发阶段数据库建议：

- SQLite

后续正式阶段可切换：

- PostgreSQL

## 9.2 Token 机制升级

当前已经使用：

- `JWT Access Token`

后续建议升级为：

- `Refresh Token`

原因：

- 可签名
- 可设过期时间
- 可扩展刷新机制
- 更适合前后端分离系统

当前使用组件：

- `python-jose`

后续可继续沿用，不必更换。

## 9.3 权限模型升级

后续建议从“板块级 RBAC 权限”升级为：

1. 菜单权限
2. 接口权限
3. 数据范围权限
4. 项目范围权限

也就是说未来可形成：

- 用户 -> 角色 -> 权限
- 角色 -> 菜单 / 接口 / 数据范围

## 9.4 缓存与登录态

后续建议把会话控制增强到：

- Redis

好处：

- 支持服务重启后保留会话
- 支持统一失效控制
- 支持多实例部署

## 10. 结论

当前登录与权限管理部分，采用的是：

- FastAPI 原生路由能力
- FastAPI 依赖注入机制 `Depends`
- Header 中的 `Bearer JWT`
- Pydantic 参数校验
- `python-jose` 生成与解析 JWT
- `passlib` 做密码哈希与校验
- SQLite 持久化用户体系
- 基础 RBAC 模型
- 自定义认证服务 `AuthService`
- 自定义用户服务 `user_service`
- 自定义菜单权限服务 `menu_service`

这不是依赖某个现成认证框架的实现，而是一套“适合当前阶段、比纯轻量方案更规范”的自定义方案。

对于当前阶段来说，这个方案是合理的，因为它已经足够支持：

- 用户名/手机号登录
- JWT 登录态验证
- SQLite 持久化用户与角色
- 板块级 RBAC 权限控制
- 用户权限配置
- 前后端菜单联动

但在进入正式开发中后期时，建议逐步升级到：

- PostgreSQL 持久化用户体系
- Refresh Token
- Redis 会话管理
- 更完整的 RBAC 权限模型
